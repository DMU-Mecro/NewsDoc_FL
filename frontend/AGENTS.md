# Git 브랜치 & 커밋 명명 규칙

## 브랜치 네이밍 규칙

### 형식

```
type/short-description-author
```

### type 종류

| type       | 설명                       |
| ---------- | -------------------------- |
| `feature`  | 기능 개발                  |
| `fix`      | 버그 수정                  |
| `hotfix`   | 운영 긴급 수정             |
| `refactor` | 리팩토링                   |
| `design`   | UI/디자인 작업             |
| `docs`     | 문서 작업                  |
| `test`     | 테스트 코드                |
| `chore`    | 설정, 패키지, 잡일         |
| `release`  | 배포 준비                  |
| `infra`    | 서버, Docker, Nginx, CI/CD |

### 작성 규칙

- 전부 소문자
- 띄어쓰기 금지
- 단어 구분은 `-` (하이픈)
- 너무 길지 않게
- 기능 단위가 보이게 작성

### 예시

```text
feature/login-api
feature/user-profile-edit
fix/signup-validation
refactor/api-interceptor
design/main-chart-ui
```

## 커밋 메시지 규칙

### 형식

```
type: 내용
```

### type 종류

| type       | 설명                                                             |
| ---------- | ---------------------------------------------------------------- |
| `feat`     | 기능 (새로운 기능)                                               |
| `fix`      | 버그 수정                                                        |
| `refactor` | 리팩토링                                                         |
| `design`   | CSS 등 사용자 UI 디자인 변경                                     |
| `comment`  | 필요한 주석 추가 및 변경                                         |
| `style`    | 스타일 (코드 형식, 세미콜론 추가: 비즈니스 로직에 변경 없음)     |
| `docs`     | 문서 수정 (문서 추가, 수정, 삭제, README)                        |
| `test`     | 테스트 (테스트 코드 추가, 수정, 삭제: 비즈니스 로직에 변경 없음) |
| `chore`    | 기타 변경사항 (빌드 스크립트 수정, assets, 패키지 매니저 등)     |
| `init`     | 초기 생성                                                        |
| `rename`   | 파일 혹은 폴더명을 수정하거나 옮기는 작업만 한 경우              |
| `remove`   | 파일을 삭제하는 작업만 수행한 경우                               |

### 커밋 단위 규칙

- **기능 하나 완성 → 즉시 커밋** (작업 뭉텅이로 한 번에 올리지 않습니다)
- 여러 기능을 동시에 수정했더라도 **기능별로 나눠서 따로 커밋**합니다
- `git add` 할 때 관련 파일만 선택하여 스테이징합니다

```bash
❌ 나쁜 예 — 파일 왕창 올리기
git add .
git commit -m "feat: 여러 기능 작업"

✅ 좋은 예 — 기능별로 나눠서 올리기
git add src/api/index.js
git commit -m "feat: Axios 인터셉터 JWT 자동 주입 로직 추가"

git add src/pages/Login.jsx
git commit -m "design: 로그인 페이지 UI 구현"
```

### 본문 작성 규칙

- 본문은 `어떻게` 보다 `무엇을`, `왜`를 설명합니다.
- 본문에 여러 줄의 메시지를 작성할 땐 `-`로 구분합니다.
- **커밋 메시지는 전부 한국어로 작성합니다.**
- 커밋 메시지는 **무슨 작업을 했는지 바로 알 수 있게** 작성합니다.
- 가능하면 "추가", "수정", "제거"처럼 명확하게 작성합니다.
- **본문은 세세하게**: 변경한 파일/함수/로직을 구체적으로 명시합니다.
- 추상적인 표현 금지 (❌ "기능 수정" → ✅ "토큰 만료 시 로그인 페이지로 리다이렉트 하도록 수정")

## AI 에이전트 지시사항

> AI는 아래 규칙을 기반으로 브랜치명과 커밋 메시지를 자동 생성합니다.

### 브랜치명 생성 규칙
1. 작업 내용을 파악하여 적절한 `type`을 선택합니다.
2. 핵심 기능을 2~4단어 이내의 소문자 영어로 표현합니다.
3. 단어는 `-`로 구분하며, 필요 시 작성자 이니셜을 끝에 붙입니다.
4. 브랜치명 전체는 소문자만 사용합니다.

### 커밋 메시지 생성 규칙
1. 제목은 `type: 내용` 형식으로 작성합니다.
2. **제목과 본문 모두 한국어로 작성합니다.**
3. **`git add .` 사용 금지** — 반드시 관련 파일만 골라서 스테이징하도록 스크립트를 제공합니다.
4. **기능 단위로 커밋을 나눠서** 각각 메시지를 생성합니다.
5. 본문은 **세세하게** 작성합니다. (변경된 파일명, 함수명, 구체적인 로직 등 포함)
